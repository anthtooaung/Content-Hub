<objective>
Fix the broken data layer by adding a Content model to the Prisma schema, then wire the save flow from the generate page so users can persist generated content to the database.

Purpose: The history API uses prisma.content.findMany() and prisma.content.create(), but the Prisma schema has no Content model -- only content_generations, ai_responses, and saved_posts. This means the dashboard always shows empty and saves silently fail. Adding the Content model and a Save button on the generate page completes the save-to-dashboard pipeline.

Output: A working save flow: user generates content, clicks Save, content persists, and appears on the dashboard.
</objective>

  <files>prisma/schema.prisma</files>
  <action>
Add a Content model to prisma/schema.prisma after the existing saved_posts model. The model must match the fields the history API expects:

- id: String, @id @default(cuid())
- userId: String (maps to users.id via relation)
- platform: String (store as plain String, not enum -- the API sends string values like "TikTok", "Instagram", "Facebook")
- tone: String? (optional)
- post: String @db.Text
- hashtags: String[] (PostgreSQL array)
- caption: String? @db.Text
- callToAction: String? @db.Text
- createdAt: DateTime @default(now())
- user relation: users @relation(fields: [userId], references: [id], onDelete: Cascade)

Add @@index([userId]) and @@index([createdAt(sort: Desc)]) for query performance.

Also add the Content relation to the users model: `contents Content[]` in the relations section.

After editing the schema, run `npx prisma db push` to sync the new model to the database. Then run `npx prisma generate` to regenerate the Prisma client. Verify no errors from either command.

Important: Use String type for platform (not the platform_type enum) because the existing API sends string values like "TikTok" with capital letters, and the dashboard interface expects strings. The enum uses lowercase values which would cause mismatches.
